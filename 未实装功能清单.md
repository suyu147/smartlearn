# SmartLearn 未实装功能清单

> 本文档梳理了项目中已搭建基本架构但缺少关键代码导致无法实装的功能。
> 每项功能均标注了涉及的文件路径、当前状态、缺失部分和建议修复方案。

---

## 1. 视频生成功能 — 仅生成脚本，未接入视频生成API

**当前状态**：`resource-prompts.json` 中定义了 `video` 类型的提示词（生成教学视频脚本大纲），API 路由 `/api/generate/resources` 也支持 `video` 类型，但实际只通过 LLM 生成文字脚本，**未接入任何视频生成API**（如 SeeDance、D-ID 等）。

**涉及文件**：
- `lib/prompts/resource-prompts.json` — video 提示词仅生成"脚本大纲"
- `app/api/generate/resources/route.ts` — video 类型走的是和 document 一样的 `streamLLM` 文字生成
- `components/resources/resource-grid.tsx` — video 和 reading 类型只显示纯文本 `<pre>` 渲染
- `lib/types/resource.ts` — `ResourceType` 包含 `'video'`，但无视频URL字段
- `lib/store/media-generation.ts` — 定义了 `MediaGenerationTask`（含 `type: 'video'`、`resultUrl`），但从未被调用

**缺失部分**：
- 视频生成 API 接入（SeeDance / D-ID / 其他）
- 视频生成后的 URL 回填到 Resource 对象
- 前端视频播放器组件（`video-player.tsx` 在计划书中列出但不存在）
- `MediaGenerationTask` Store 的实际使用逻辑

**建议修复**：
1. 在 `lib/` 下创建 `lib/video/` 模块，封装视频生成 API 调用
2. 修改 `app/api/generate/resources/route.ts` 中 video 类型的处理：先用 LLM 生成脚本，再调用视频API生成视频
3. 创建 `components/resources/video-player.tsx` 视频播放器组件
4. 将视频 URL 存入 `Resource.metadata.videoUrl`，前端通过播放器展示

**缺失部分**：

**建议修复**：
1. 重写 `MindmapViewer` 组件，引入 `markmap-view` 的 `Markmap` 类
2. 将 LLM 生成的 Markdown 文本传给 Markmap 进行渲染

**当前状态**：`/api/web-search` 路由已完整搭建，包含 LLM 查询重写、SSRF 防护等，但核心搜索函数 `searchWithTavily` **返回空结果**，未实际调用 Tavily API。

- `app/api/web-search/route.ts` — 完整的请求处理逻辑，但依赖空的 `searchWithTavily`

**缺失部分**：
- Tavily API 的实际 HTTP 调用
- 搜索结果的解析和格式化

**建议修复**：
1. 在 `searchWithTavily` 中实现 Tavily API 调用：
   ```typescript
   const response = await fetch('https://api.tavily.com/search', {
     method: 'POST',
     headers: { 'Content-Type': 'application/json' },
     body: JSON.stringify({ query: options.query, api_key: options.apiKey }),
   });
   const data = await response.json();
   return { answer: data.answer, sources: data.results, ... };
   ```

- `lib/server/resolve-model.ts` — 未读取 `apiSecret`

**缺失部分**：
- 讯飞星火旧版 WebSocket API 的签名认证逻辑
- `apiSecret` 从设置页到 API 调用链的传递

**建议修复**：
1. 如果仅使用 OpenAPI 兼容模式，可移除 API Secret 相关代码

## 13. 设置页面的 AI 配置未传递到 API 调用

**当前状态**：设置页面可以配置 AI 提供商、模型、API Key 等，这些值保存在 Zustand Store（localStorage）中，但**各 API 路由（profile/chat、generate/resources、path/plan、tutor/chat）均从服务端环境变量读取配置**，完全忽略了客户端设置。

**涉及文件**：
- `app/settings/page.tsx` — 配置界面
- `lib/store/settings.ts` — 保存配置到 localStorage
- `app/api/profile/chat/route.ts` — 从 `process.env` 读取配置
- `app/api/generate/resources/route.ts` — 从 `process.env` 读取配置
- `app/api/path/plan/route.ts` — 从 `process.env` 读取配置
- `app/api/tutor/chat/route.ts` — 从 `process.env` 读取配置

**缺失部分**：
- 前端将设置页配置传递给 API 请求（通过 headers 或 body）
- API 路由优先使用客户端传递的配置，环境变量作为 fallback

**建议修复**：
1. 前端在调用 API 时，将 `providerId`、`modelId`、`apiKey` 等通过请求体或 headers 传递
2. API 路由优先使用客户端传递的配置
3. 参考 `/api/chat/route.ts` 中 `resolveModelFromHeaders` 的实现模式
